# claude_settings

Claude Code(Anthropic CLI 에이전트)용 커스텀 Skill/Workflow 시스템.

## 개념

### Skills는 라우팅만 한다

Skill은 하나(`router`)만 존재하고, 그 역할은 사용자 요청의 의도를 파악하여 적절한 워크플로우로 분기하는 것뿐이다. 실제 작업 로직은 전부 워크플로우에 있다.

### 계층적 컨텍스트 주입

프롬프트로 직접 명령하지 않아도 정해진 워크플로우가 실행된다. 이것이 가능한 이유는 에이전트가 시작 시 컨텍스트를 계층적으로 주입받기 때문이다.

```
CLAUDE.md                    ← "모든 요청에 router skill을 먼저 써라"
  └─ router/SKILL.md         ← "요청 분석 후 해당 workflow.md를 읽고 따라라"
       └─ {workflow}/workflow.md  ← 실제 작업 절차 정의
```

CLAUDE.md가 router 호출을 강제하고, router가 워크플로우를 불러오면서 에이전트는 사용자가 매번 "PDCA 방식으로 해줘"라고 말하지 않아도 항상 동일한 프로세스를 거친다.

## 디렉토리 구조

```
.claude/
├── skills/
│   └── router/SKILL.md              # 라우팅 전용 스킬
└── workflows/
    ├── code-work/workflow.md         # 코드 작업 (PDCA + TDD)
    ├── plan-review/workflow.md       # Plan 검증 (서브 에이전트)
    ├── pdca-check/workflow.md        # Check 단계 (서브 에이전트)
    ├── code-review/workflow.md       # Code Review (서브 에이전트)
    ├── documentation/workflow.md     # 문서 작성
    └── claude-md-review/workflow.md  # CLAUDE.md 품질 검토

claude_docs/
└── claude_md_guide.md               # CLAUDE.md 작성 가이드 (리뷰 기준)

claude_output_docs/                  # 워크플로우 산출물 출력 경로
└── claude_md_review.md              # CLAUDE.md 리뷰 결과 (누적)

claude_task/<task_id>/               # 코드 작업 산출물 (YYYYMMDD-순번)
├── plan/
│   ├── plan.md
│   └── plan_review.md
├── do/do.md
├── check/check.md
├── act/act.md
└── review/review.md
```

## PDCA 워크플로우 상세

코드 작업 요청이 들어오면 `code-work` 워크플로우가 실행된다. 6단계로 구성된다.

### Step 0 — 요청 확인

작업 시작 전 에이전트가 이해한 내용(작업 목표, 예상 범위, 제약 사항)을 사용자에게 보여주고 확인을 받는다. 사용자가 확인해야만 다음 단계로 진행한다.

### Step 1 — Plan

사용자 요청을 분석하고 작업 범위를 정의한다. `plan.md`에 기록한다.

- 요청 요약
- 영향 범위 (변경될 파일/모듈)
- 접근 방식

### Step 1.5 — Plan Review (서브 에이전트, 최대 3회)

메인 에이전트가 직접 검증하지 않는다. **서브 에이전트**로 `plan-review` 워크플로우를 실행하여 `plan.md`를 독립적으로 검증한다.

검증 관점:
- 요청 부합성 / 범위 명확성 / 실현 가능성
- 영향 범위 파악 / 접근 방식 적절성
- 오버 엔지니어링 방지

결과는 `plan_review.md`에 **누적** 기록된다. 이전 사이클의 피드백을 삭제하지 않고 구분선(`---`)으로 누적하므로, 다음 사이클의 서브 에이전트가 "이전에 어떤 지적이 있었는지"를 파악할 수 있다.

```
승인      → 즉시 Do 단계로 진행
리플랜 필요 → 피드백 확인 후 plan.md 수정 → 재검증
3회 초과   → 빨간색 경고 출력, 사용자 수동 개입 요청
```

> 서브 에이전트는 plan.md를 직접 수정하지 않는다. 검증과 피드백만 수행한다.

### Step 2 — Do (TDD Red)

테스트를 먼저 작성한다. 이 시점에서는 구현 코드가 없으므로 테스트는 반드시 실패해야 한다.

- 테스트 파일 형식: `<filename>.py.test` (pytest)
- `do.md`에 기록: 작성한 테스트 목록, 각 테스트의 검증 의도

### Step 3 — Check (서브 에이전트)

**서브 에이전트**로 `pdca-check` 워크플로우를 실행한다. Plan Review와 달리 이 서브 에이전트는 **직접 수정**까지 수행한다.

수행 내용:
1. `plan.md`, `do.md`, 테스트 파일 읽기
2. pytest 실행 및 결과 확인
3. 코드 품질 점검 (메모리 효율성 / 네이밍 / 오버 엔지니어링 / 미사용 코드 / 근본 원인 해결)
4. 문제 발견 시 직접 수정
5. `check.md`에 결과 기록

### Step 4 — Act (TDD Green + Refactor)

Check가 완료된 코드를 기반으로 본 구현에 들어간다.

- 코드 파일 최상단에 `# ## <task_id>` 주석 추가
- `act.md`에 기록: 작업 대상 파일, 변경 내용 요약, 최종 테스트 결과

### Step 5 — Code Review (서브 에이전트, 최대 3회)

**서브 에이전트**로 `code-review` 워크플로우를 실행하여 최종 코드를 검증하고 커밋 가능 여부를 판단한다.

검증 관점:
- 코드 품질: 메모리 효율성 / 네이밍 / 오버 엔지니어링 / 미사용 코드 / 에지 케이스
- 설계 및 구조: Plan과의 일치 여부 / 근본 원인 해결 / 책임 분리 / 의존성 / 확장성
- 가독성 및 일관성

결과는 `review.md`에 **누적** 기록된다 (Plan Review와 동일한 방식).

```
승인(커밋 가능) → 작업 완료
수정 필요       → 수정 후 Check(Step 3)부터 재수행
3회 초과        → 빨간색 경고 출력, 사용자 수동 개입 요청
```

> 서브 에이전트는 코드를 직접 수정하지 않는다. 검증과 피드백만 수행한다.

## claude-md-review 워크플로우

`CLAUDE.md`의 품질을 `claude_docs/claude_md_guide.md` 기준으로 검토하고 문제점과 개선점을 제시한다. 코드 작업과 독립적으로 단독 실행된다.

검토 항목:
- **포함 여부**: 기술 스택, 프로젝트 구조, 명령어, 아키텍처 철학, 패턴, 도메인 용어 등
- **제외 여부**: API 키/비밀번호, linter 규칙, 프레임워크 기본 지식, 과도한 문서 등
- **길이 및 구조**: 300줄 이하, 상세 내용은 파일 참조 방식으로 분리
- **Anti-Pattern**: 금지 항목에 대안 명시, 스타일 규칙은 결정론적 도구로 위임

결과는 `claude_output_docs/claude_md_review.md`에 누적 기록된다. 이전 리뷰의 피드백과 해소 여부가 함께 쌓이므로 개선 이력을 추적할 수 있다.

## 서브 에이전트 사용 원칙

| 워크플로우 | 실행 주체 | 직접 수정 | 역할 |
|-----------|---------|---------|------|
| plan-review | 서브 에이전트 | X | plan.md 검증 + 피드백 |
| pdca-check | 서브 에이전트 | O | 테스트 실행 + 품질 점검 + 수정 |
| code-review | 서브 에이전트 | X | 최종 코드 검증 + 커밋 가능 여부 판단 |

서브 에이전트를 쓰는 이유:
- 메인 컨텍스트 윈도우를 검증 로직으로 소모하지 않는다
- "작성자"와 "리뷰어"가 분리되어 독립적인 관점에서 판단할 수 있다

## 검증 루프

Plan Review와 Code Review는 최대 3회 반복되는 검증 루프를 가진다.

```
[서브 에이전트 실행]
       │
       ├─ 승인 → 다음 단계 진행
       │
       └─ 수정 필요
              │
              ├─ 1~2회차: 피드백 기반 수정 → 재검증
              │
              └─ 3회 초과: 실패 출력, 사용자 개입 요청
```

각 검증 결과는 누적 기록되어 이전 사이클의 피드백과 해소 여부를 다음 서브 에이전트가 확인할 수 있다.

## TDD 원칙

| 단계 | 설명 |
|------|------|
| Red (Do) | 실패하는 테스트를 먼저 작성 |
| Green (Act) | 테스트를 통과하는 최소한의 코드 작성 |
| Refactor (Act) | 통과 후 코드 정리 (테스트는 계속 통과해야 함) |
