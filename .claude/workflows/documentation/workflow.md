# Documentation Workflow

일반적인 문서화 요청에 대응하는 워크플로우이다.
README, API 문서, 아키텍처 문서, 가이드 등 모든 유형의 문서 작성/갱신을 다룬다.

## 문서 출력 경로

```
docs/claude/<유저 요청에 맞는 세부 경로>/
```

문서 파일은 `docs/claude/` 하위에 작업 내용에 맞게 적절한 세부 경로를 구성하여 생성한다.
예시:
- API 문서 → `docs/claude/api/user-api.md`
- 아키텍처 문서 → `docs/claude/architecture/overview.md`
- 가이드 → `docs/claude/guides/setup.md`

## 작업 기록 경로

```
docs/claude/_reviews/
├── docs_log.md       # 작업 기록 (어떤 문서를 작성/갱신했는지)
└── docs_review.md    # 검증 결과
```

## 터미널 출력 규칙

각 스텝 진입 시 현재 단계를 터미널에 출력한다:
```
echo -e "\033[36m[DOCS STEP N] <단계명>\033[0m"
```

## Steps

### 0. 요청 확인
- 유저의 문서화 요청을 정리하여 이해한 내용이 맞는지 확인을 받는다.
- 정리 항목: 문서 유형, 대상 범위, 독자(누구를 위한 문서인지), 세부 경로(`docs/claude/` 하위)
- 유저가 확인해야 다음 단계로 진행한다.
- **승인 직후**, 생성될 문서 파일 경로를 초록색으로 터미널에 출력한다:
  - `echo -e '\033[32m[DOCS] 생성 경로: docs/claude/<세부경로>/<파일명>.md\033[0m'`
  - 여러 파일인 경우 각 경로를 한 줄씩 출력한다.

### 1. 조사
- 문서화 대상 코드/시스템을 탐색하여 필요한 정보를 수집한다.
- 기존 문서가 있다면 읽고 현행화 필요 여부를 파악한다.
- `docs/claude/_reviews/docs_log.md`에 기록: 조사 결과, 문서화 대상 목록, 기존 문서 상태

### 2. 작성
- 조사 결과를 기반으로 문서를 작성하거나 갱신한다.
- `docs/claude/_reviews/docs_log.md`에 추가 기록: 작성/갱신한 문서 파일 경로 목록, 주요 내용 요약

#### 작성 원칙
- **간결 우선**: 기본적으로 간결하게 작성한다. 장황한 문서는 지양한다.
- **필요시 구체적으로**: 복잡한 로직, 비직관적인 설계, 주의사항 등은 구체적으로 작성한다.
- **판단 기준**: 간결하게 쓸지 구체적으로 쓸지 스스로 판단하되, 애매한 경우 유저에게 물어본다.
- **마크다운 적극 활용**: 헤딩, 목록, 코드 블록, 테이블, 링크 등 마크다운 요소를 활용하여 가독성을 높인다.

### 3. 문서 검증 (최대 3회 반복)
- 서브 에이전트로 문서 품질을 검증한다. 메인 컨텍스트에서 직접 실행하지 않는다.
  - 예시: `Task(subagent_type="general-purpose", prompt="docs/claude/_reviews/docs_log.md를 읽고 작성된 문서 파일을 확인한 뒤, .claude/workflows/documentation/workflow.md의 검증 체크리스트에 따라 문서 품질을 검증하고 결과를 docs/claude/_reviews/docs_review.md에 기록해줘. 현재 리뷰 사이클: N/3회차")`

- **승인** → 초록색으로 승인 출력 후 완료
  - `echo -e "\033[32m[APPROVED] 문서 검증 승인 완료\033[0m"`
- **수정 필요** → 피드백을 기반으로 문서 수정 후 재검증
- **최대 반복 횟수는 3회**이다. 3회 후에도 승인되지 않으면 빨간색으로 실패를 알린다.
  - `echo -e "\033[31m[DOCS FAILED] 3회 문서 검증 반복 후에도 승인되지 않았습니다. 수동 개입이 필요합니다.\033[0m"`

## 검증 체크리스트

서브 에이전트는 `docs/claude/_reviews/docs_review.md`에 다음 형식으로 기록한다.

#### 누적 기록 규칙
- 파일 최상단 제목(`# Documentation Review - <task_id>`)은 최초 1회만 작성한다.
- 각 사이클은 `---` 구분선으로 시작하여 파일 하단에 누적 추가한다.
- **이전 사이클의 내용은 절대 삭제/수정하지 않는다.**
- 2회차 이상에서는 체크리스트 앞에 "이전 사이클 피드백 해소 여부" 인용 블록을 배치하여, 이전 피드백이 어떻게 반영되었는지 명시한다.

#### 템플릿

```markdown
# Documentation Review - <task_id>

---

## 사이클 1/3

### 검증 대상 문서
- path/to/doc1.md
- path/to/doc2.md

### 체크리스트

#### 정확성
- [ ] 기술적 정확성: 코드/시스템의 실제 동작과 문서 내용이 일치
- [ ] 코드 예시: 포함된 코드 예시가 실제로 동작 가능
- [ ] 버전 정합성: 현재 코드 버전과 문서 내용이 일치

#### 완성도
- [ ] 범위 충족: 요청된 문서화 범위를 빠짐없이 다룸
- [ ] 필수 섹션: 문서 유형에 맞는 필수 섹션이 포함됨
- [ ] 전제 조건: 독자가 알아야 할 사전 지식/환경이 명시됨

#### 가독성 및 작성 품질
- [ ] 구조: 논리적 흐름과 계층 구조가 명확
- [ ] 용어 일관성: 동일 개념에 동일 용어 사용
- [ ] 간결성: 불필요한 반복이나 장황한 설명 없음
- [ ] 적절한 상세도: 간결해야 할 부분은 간결하고, 구체적이어야 할 부분은 구체적
- [ ] 마크다운 활용: 헤딩, 목록, 코드 블록, 테이블 등을 적절히 활용하여 가독성 확보

### 피드백
(미통과 항목에 대해 구체적인 위치와 개선 방향 기술)

### 종합 판정: **승인** / **수정 필요**

---

## 사이클 2/3

> **이전 사이클 피드백 해소 여부**
> - [항목명]: 해소됨/미해소. (근거)

### 검증 대상 문서
(동일 문서 목록)

### 체크리스트
(동일 체크리스트)

### 피드백
(해당 사이클의 피드백)

### 종합 판정: **승인** / **수정 필요**
```

- 통과한 항목은 `[x]`, 미통과 항목은 `[ ]`로 표기
- 미통과 항목이 하나라도 있으면 **수정 필요**로 판정

**주의: 서브 에이전트는 문서를 직접 수정하지 않는다. 검증과 피드백만 수행한다.**
