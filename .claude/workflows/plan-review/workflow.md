# Plan Review Workflow

PDCA 사이클의 Plan 단계에서 작성된 plan.md를 검증한다.
본 워크플로우는 서브 에이전트로 실행되며, Plan의 품질을 보장하는 게이트 역할을 한다.
직접 수정하지 않고, 문제점과 개선방안을 정리하여 보고한다.

## 산출물 경로

```
claude_task/<task_id>/plan/
├── plan.md          # 원본 계획
└── plan_review.md   # 검증 결과
```

## 검증 절차

### 1. 컨텍스트 파악
- `claude_task/<task_id>/plan/plan.md` — 작성된 계획 내용

### 2. 검증 관점
다음 관점에서 plan.md를 평가한다:

- **요청 부합성**: 유저의 원래 요청 의도를 정확히 반영하고 있는가
- **범위 명확성**: 작업 범위가 구체적이고 모호하지 않은가
- **실현 가능성**: 현재 프로젝트 구조와 기술 스택에서 실현 가능한 계획인가
- **영향 범위 파악**: 변경에 의해 영향받는 파일/모듈이 빠짐없이 식별되었는가
- **접근 방식 적절성**: 제안된 접근 방식이 문제에 대한 합리적인 해법인가
- **오버 엔지니어링 방지**: 요청 대비 과도한 설계가 포함되어 있지 않은가

### 3. 결과 기록
`plan_review.md`에 다음 형식으로 기록한다.

#### 누적 기록 규칙
- 파일 최상단 제목(`# Plan Review - <task_id>`)은 최초 1회만 작성한다.
- 각 사이클은 `---` 구분선으로 시작하여 파일 하단에 누적 추가한다.
- **이전 사이클의 내용은 절대 삭제/수정하지 않는다.**
- 2회차 이상에서는 체크리스트 앞에 "이전 사이클 피드백 해소 여부" 인용 블록을 배치하여, 이전 피드백이 어떻게 반영되었는지 명시한다.

#### 템플릿

```markdown
# Plan Review - <task_id>

---

## 사이클 1/3

### 체크리스트

#### 계획 품질
- [ ] 요청 부합성: 유저의 원래 요청 의도를 정확히 반영
- [ ] 범위 명확성: 작업 범위가 구체적이고 모호하지 않음
- [ ] 실현 가능성: 현재 프로젝트 구조와 기술 스택에서 실현 가능

#### 분석 완성도
- [ ] 영향 범위 파악: 영향받는 파일/모듈이 빠짐없이 식별됨
- [ ] 접근 방식 적절성: 문제에 대한 합리적인 해법

#### 설계 균형
- [ ] 오버 엔지니어링 방지: 요청 대비 과도한 설계 없음

### 피드백
(미통과 항목에 대해 문제점과 개선방안 기술)

#### 1. [미통과 항목명]
- 문제점: ...
- 개선방안: ...

### 종합 판정: **승인** / **리플랜 필요**

---

## 사이클 2/3

> **이전 사이클 피드백 해소 여부**
> - [항목명]: 해소됨/미해소. (근거)

### 체크리스트
(동일 체크리스트)

### 피드백
(해당 사이클의 피드백)

### 종합 판정: **승인** / **리플랜 필요**
```

- 통과한 항목은 `[x]`, 미통과 항목은 `[ ]`로 표기
- 미통과 항목이 하나라도 있으면 **리플랜 필요**로 판정

**주의: 서브 에이전트는 plan.md를 직접 수정하지 않는다. 검증과 피드백만 수행한다.**
